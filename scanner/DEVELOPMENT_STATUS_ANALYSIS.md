# Анализ текущего состояния разработки системы

## 1. Анализ текущего состояния разработки системы

На момент начала производственной практики проект «Интеллектуальная система обучения иностранным языкам» находился на стадии завершенного проектирования и первичной технической подготовки. В рамках курсового проектирования был заложен фундамент, позволяющий перейти к непосредственной программной реализации модулей веб-приложения.

### 1.1 Этап проектирования

В ходе курсового проектирования были выполнены следующие работы:

- **Анализ требований**: Определение функциональных и нефункциональных требований системы на основе анализа потребностей пользователей и рынка образовательных технологий
- **Проектирование архитектуры**: Разработка микросервисной архитектуры с разделением на фронтенд, бэкенд и AR-модуль
- **Моделирование данных**: Создание схемы базы данных с учетом взаимосвязей между пользователями, словарями, уроками и AR-метками
- **Проектирование API**: Разработка интерфейсов для взаимодействия между компонентами системы
- **Прототипирование интерфейса**: Создание макетов пользовательского интерфейса с учетом принципов UX для AR-приложений

### 1.2 Техническая подготовка

Проведена первичная техническая подготовка, включающая:

- **Настройка окружения разработки**: Установка и конфигурация необходимых инструментов и библиотек
- **Создание структуры проекта**: Организация файловой структуры в соответствии с принципами Django-разработки
- **Реализация базовых моделей**: Создание моделей базы данных и их миграций
- **Настройка системы контроля версий**: Организация рабочего процесса с использованием Git
- **Подготовка тестовых данных**: Создание наборов данных для тестирования функциональности системы

## 2. Выявление проблемных мест и потенциальных сложностей

В ходе детального анализа проектных материалов и проведения предварительных тестов были определены критические зоны риска, которые могут оказать существенное влияние на сроки и качество реализации системы.

### 2.1 Технологические сложности

#### 2.1.1 Ограничения PWA

В отличие от нативных приложений, Progressive Web Apps имеют ограниченный доступ к аппаратному ускорению графики, что создает определенные challenges для реализации AR-функциональности. Основная сложность заключается в обеспечении плавности видеопотока (не менее 30 кадров в секунду) при одновременной работе скриптов распознавания, что может приводить к «зависанию» интерфейса и негативному пользовательскому опыту.

**Механизм возникновения проблемы:**
1. JavaScript-интерпретатор выполняет обработку видеокадров и распознавание объектов в едином потоке
2. При высокой нагрузке主线程 блокируется, что приводит к задержкам в отрисовке интерфейса
3. Пользователь замечает «подвисания» и нестабильность работы приложения

**Потенциальные решения:**
- **Использование Web Workers**: Перенос обработки видео и распознавания объектов в фоновые потоки, освобождая主线程 для отрисовки интерфейса
- **Оптимизация алгоритмов**: Уменьшение количества вычислений за один кадр, использование инкрементного анализа
- **Адаптивный режим**: Автоматическое снижение частоты кадров или качества обработки при низкой производительности устройства
- **WebAssembly**: Перевод критических участков кода на WebAssembly для увеличения скорости выполнения

#### 2.1.2 Вычислительные требования YOLO8

Использование YOLO8 для реального времени требует значительных вычислительных мощностей, особенно при обработке высококачественных видео. Существует риск, что на смартфонах среднего и бюджетного сегментов (с процессорами Snapdragon 600-серии или аналогами) время детекции объекта превысит допустимый порог (500 мс), что сделает использование AR-модуля некомфортным и может привести к отказу пользователей от его использования.

**Механизм возникновения проблемы:**
1. YOLO8 выполняет анализ каждого кадра, что требует интенсивных вычислений
2. На устройствах с ограниченными ресурсами GPU обработка занимает больше времени
3. Пользователь должен ждать завершения распознавания, что разрушает непрерывность пользовательского опыта

**Потенциальные решения:**
- **Гибридный подход**: Реализация легковесной версии модели на клиенте для быстрого предварительного распознавания и полного анализа на сервере
- **Оптимизация модели**: Применение quantization (снижение точности весов) и pruning (удаление незначительных нейронных связей) для уменьшения размера модели и увеличения скорости выполнения
- **Edge Computing**: Использование ближайших к пользователю серверов для обработки данных, что уменьшает задержки при передаче
- **Адаптивная обработка**: Автоматическое подбор параметров модели в зависимости от мощности устройства
- **Инкрементное распознавание**: Использование результатов предыдущего кадра для ускорения обработки текущего

#### 2.1.3 Зависимость точности распознавания от внешних факторов

Точность распознавания объектов с помощью YOLO8 сильно зависит от ряда внешних факторов: уровня освещенности, угла наклона камеры, степени перекрытия объектов, качества изображения и наличия препятствий. В условиях неоптимальных параметров может наблюдаться высокий процент ложноположительных и ложноотрицательных срабатываний, что снижает доверие пользователя к системе.

**Механизм возникновения проблемы:**
1. Низкая освещенность приводит к потере деталей объекта
2. Неверный угол наклона камеры изменяет внешний вид объекта
3. Перекрытие объектов затрудняет их идентификацию
4. Шум на изображении干扰рует работе нейросети

**Потенциальные решения:**
- **Адаптивная компенсация освещенности**: Использование алгоритмов обработки изображения для коррекции экспозиции и контраста
- **Фильтры постобработки**: Применение статистических фильтров для исключения ложноположительных срабатываний
- **Система обратной связи**: Сбор данных о неудачных распознаваниях для последующей дообучения модели
- **Мультипозиционная проверка**: Анализ нескольких кадров для увеличения надежности распознавания
- **Контекстная проверка**: Использование информации о среде (например, кухня, зал) для уточнения вероятности присутствия определенных объектов

### 2.2 Инфраструктурные сложности

#### 2.2.1 Размер ML-библиотек

ML-библиотеки (TensorFlow, PyTorch, OpenCV) имеют большой объем (от 1 до 5 ГБ) и специфические требования к системным библиотекам ОС (например, CUDA для GPU-ускорения). Это создает значительные сложности при сборке легковесных Docker-контейнеров и может привести к конфликтам версий при развертывании на различных серверах.

**Механизм возникновения проблемы:**
1. Стандартные базовые образы Docker не содержат необходимых системных библиотек
2. Установка всех зависимостей увеличивает размер контейнера до неуправляемых масштабов
3. Версионные конфликты между библиотеками могут привести к сбоям в работе
4. Развертывание большого контейнера занимает значительное время и ресурсы

**Потенциальные решения:**
- **Специализированные базовые образы**: Использование официальных базовых образов для ML-приложений, которые уже содержат необходимые зависимости
- **Оптимизация зависимостей**: Удаление ненужных компонентов и модулей из библиотек
- **Разделение контейнеров**: Создание отдельных контейнеров для API (лёгкий) и ML-обработки (тяжелый)
- **Layered approach**: Организация Docker-образов с использованием слоев для более эффективного кэширования
- **Serverless решение**: Использование серверныхless платформ для ML-обработки, что убирает проблемы с развертыванием

#### 2.2.2 Сложности с Docker-контейнерами

Создание и управление Docker-контейнерами для ML-приложений представляет собой сложную задачу. Особые трудности возникают при настройке среды для работы с GPU и при обеспечении совместимости между различными версиями библиотек.

**Механизм возникновения проблемы:**
1. Требования к версиям CUDA и cuDNN для GPU-ускорения
2. Конфликты между системными библиотекой и ML-библиотеками
3. Нужность постоянного обновления контейнеров при обновлении библиотек
4. Сложности с отладкой проблем внутри контейнеров

**Потенциальные решения:**
- **Использование готовых контейнеров**: Использование официальных контейнеров от NVIDIA и поставщиков ML-библиотек
- **Автоматизированная сборка**: Создание скриптов для автоматической сборки и обновления контейнеров
- **Оркестрация контейнеров**: Использование Kubernetes для управления контейнерами в производственной среде
- **Мониторинг контейнеров**: Внедрение системы мониторинга для отслеживания состояния контейнеров

#### 2.2.3 Консистентность данных между PWA и облачной БД

Обеспечение консистентности данных между локальным хранилищем PWA (IndexedDB) и облачной базой данных PostgreSQL при нестабильном интернет-соединении является сложной задачей. Пользователи могут добавлять новые слова в словарь или завершать уроки офлайн, и эти данные должны быть корректно синхронизированы с сервером при восстановлении соединения.

**Механизм возникновения проблемы:**
1. Пользователь работает с приложением офлайн, изменяя данные
2. Одновременно другой пользователь (или та же самая учетная запись с другого устройства) изменяет одни и те же данные
3. При восстановлении соединения возникает конфликт, который требует разрешения
4. Неправильное разрешение конфликта может привести к потере данных или неконсистентному состоянию

**Потенциальные решения:**
- **Система кэширования и синхронизации**: Реализация двухсторонней синхронизации данных с контролем версий
- **Алгоритмы разрешения конфликтов**: Разработка правил для разрешения конфликтов, например, на основе времени модификации или приоритета изменений
- **Offline-First подход**: Разработка системы, которая приоритетно использует локальные данные и синхронизируется с сервером в фоновом режиме
- **Очередь синхронизации**: Создание очереди изменений, которая обрабатывается при восстановлении соединения
- **Контроль версий**: Использование механизмов контроля версий для отслеживания изменений и предотвращения конфликтов

#### 2.2.4 Нагрузка на канал связи

Передача видеокадров для глубокого анализа на сервер (в случае недостаточности мощностей клиента) может создать критическую нагрузку на канал связи и увеличить затраты на трафик для пользователя. Качество видео для эффективного распознавания должно быть достаточно высоким, что приводит к большим объемам данных.

**Механизм возникновения проблемы:**
1. Камера захватывает видео с высоким разрешением (720p или 1080p)
2. Данные передаются на сервер для обработки
3. На медленных соединениях (3G/4G) это приводит к задержкам
4. Пользовательский трафик растет, что может повлиять на его финансовые затраты

**Потенциальные решения:**
- **Оптимизация кодирования**: Использование современных кодеков (H.265/HEVC) для эффективного сжатия видео
- **Адаптивное качество**: Автоматическое регулирование качества видео в зависимости от скорости интернет-соединения
- **Сжатие данных**: Применение дополнительного сжатия для передачи между клиентом и сервером
- **Разбиение на пакеты**: Разделение видео на небольшие пакеты для более надежной передачи
- **Интеллектуальная передача**: Только передача кадров, где обнаружены изменения, а не всего видео
- **Локальный предварительный анализ**: Использование легковесной модели на клиенте для фильтрации кадров, которые действительно требуют глубокого анализа

### 2.3 Эксплуатационные риски

#### 2.3.1 Разрядка батареи

Длительная работа камеры в сочетании с активными вычислениями GPU/CPU приводит к быстрому разряду батареи устройства и его нагреву. Пользователи часто отказываются от использования приложений, которые существенно сокращают время автономной работы устройства.

**Механизм возникновения проблемы:**
1. Камера работает непрерывно, потребляя значительное количество энергии
2. Процесс обработки видео и распознавания объектов загружает процессор и GPU
3. Устройство нагревается, что дополнительно увеличивает потребление энергии
4. Пользователь замечает быстрый разряд батареи и перестает использовать приложение

**Потенциальные решения:**
- **Режим экономии энергии**: Автоматическое снижение частоты кадров и глубины анализа при отсутствии движения
- **Оптимизация частоты кадров**: Использование адаптивной частоты кадров в зависимости от ситуации (например, 30 кадров/сек при движении, 5 кадров/сек при статичности)
- **Настройки пользователя**: Предоставление пользователю возможности управлять уровнем потребления ресурсов через настройки
- **Фоновая обработка**: Переключение на фоновый режим при неактивности
- **Термальное управление**: Мониторинг температуры устройства и снижение нагрузки при превышении пороговых значений

#### 2.3.2 Нагрев устройства

Длительное использование камеры и выполнение интенсивных вычислений может привести к значительному нагреву устройства, что может негативно повлиять на его производительность и срок службы.

**Механизм возникновения проблемы:**
1. CPU и GPU работают на полной загрузке при обработке видео
2. Камера генерирует дополнительное тепло
3. Устройство не успевает охлаждаться
4. Система начинает снижать производительность для предотвращения перегрева

**Потенциальные решения:**
- **Термальное управление**: Реализация системы мониторинга температуры и адаптивного регулирования нагрузки
- **Оптимизация вычислений**: Уменьшение количества вычислений за единицу времени
- **Разумное ограничение времени работы**: Предоставление пользователю рекомендаций о разумном времени использования AR-функций
- **Вентиляция**: Рекомендации по использованию устройства в хорошо вентилируемых условиях

#### 2.3.3 UX-дизайн для AR-сервисов

Существует особая сложность в проектировании UX для сервисов с элементами дополненной реальности: интерфейс не должен перекрывать реальные объекты, которые пользователь сканирует, но при этом обязан предоставлять четкую и своевременную информацию о переводе, навигации и прогрессе обучения.

**Механизм возникновения проблемы:**
1. Интерфейс элементы перекрывают объект, который пользователь пытается сканировать
2. Информация отображается слишком долго или неинформативна
3. Навигационные элементы мешают восприятию реального мира
4. Пользователь испытывает когнитивную перегрузку от большого количества информации

**Потенциальные решения:**
- **Адаптивный интерфейс**: Разработка системы, которая позиционирует интерфейсные элементы в зависимости от положения распознанного объекта
- **Прозрачные слои**: Использование полупрозрачных элементов и неинтрузивных анимаций
- **Контекстная информация**: Отображение только той информации, которая прямо связана с текущим объектом
- **Тестирование UX**: Проведение тестирования с участием пользователей для оптимизации взаимодействия
- **Модульность интерфейса**: Возможность пользователю настраивать количество отображаемой информации
- **Гибкие анимации**: Использование плавных анимаций для переходов и появления информации, что снижает когнитивную нагрузку

## 3. Стратегия минимизации рисков

### 3.1 Технические меры

1. **Модульная архитектура**: Разделение системы на независимые модули с четкими интерфейсами для упрощения тестирования, отладки и масштабирования
2. **Адаптивная производительность**: Реализация интеллектуальных механизмов адаптации к мощности устройства, включая динамическое регулирование качества обработки и частоты кадров
3. **Оптимизация ML-моделей**: Использование специализированных версий YOLO8 для мобильных платформ, включая quantized и pruned версии
4. **Кэширование данных**: Реализация локального хранилища с интеллектуальной синхронизацией для уменьшения зависимости от сети
5. **Оптимизация сети**: Внедрение алгоритмов сжатия данных и адаптивного качества передачи
6. **Энергосбережение**: Разработка системы управления энергопотреблением с учетом состояния устройства и ситуации использования

### 3.2 Организационные меры

1. **План тестирования**: Разработка комплексной стратегии тестирования, включая:
   - Тестирование на различных устройствах с разной производительностью
   - Тестирование в различных условиях освещенности
   - Тестирование при разной скорости интернет-соединения
   - Пользовательские тесты для оценки UX

2. **Мониторинг производительности**: Внедрение системы мониторинга для:
   - Отслеживания времени обработки кадров
   - Анализа использования ресурсов устройства
   - Определения узких мест в коде
   - Сбора статистики об ошибках распознавания

3. **Агильная разработка**: Использование спринтов длительностью 2-3 недели с:
   - Еженедельными проверками прогресса
   - Регулярными тестированиями
   - Контролью качества кода
   - Участием пользователей в процессе разработки

4. **Документация**: Создание подробной документации, включая:
   - Архитектурные решения
   - API-спецификации
